# 批量重命名剪贴板指令中路径带双引号导致功能失效

## 摘要
修复当用户复制 `批量重命名 "E:\path"`（路径带双引号）时，程序无法执行批量重命名且无日志输出的问题。

## 背景
- 用户通过 Win32 剪贴板触发批量重命名，复制格式为 `批量重命名 <目录路径>`
- 部分用户习惯从资源管理器地址栏复制路径（含双引号），或路径本身含空格时自然带引号
- 当前 `trim()` 仅移除空白字符，双引号保留，导致 `BatchRename` 收到 `"E:\文件存储卷_1\My Kindle Content"`（含引号），`FindFirstFileW` 失败，重命名静默失效

## 需求
### 功能需求
- 触发文本 `批量重命名 "E:\path"` 中的路径双引号应被自动剥离
- 触发文本 `批量重命名 E:\path`（无引号）行为不变
- 触发文本 `批量重命名 "E:\path with spaces"` 应正确工作

### 非功能需求
- 向后兼容：已有无引号用法不受影响
- 控制台至少输出一条提示信息，让用户知晓指令已被识别

## 设计
### 架构变更
无。仅在 `WndProc` 中批量重命名触发分支添加引号剥离逻辑和一条控制台提示。

### 接口变更
无。

### 数据流
```
剪贴板文本 "批量重命名 "E:\path""
  → WndProc WM_CLIPBOARDUPDATE
  → GetClipboardText()
  → 前缀匹配 "批量重命名"
  → trim() → 移除空白 → "E:\path"          ← 新增：引号剥离
  → BatchRename(dirPath, rnpPath)           ← 现在收到正确的裸路径
```

## 实现计划
1. 在 `WndProc` lambda 中 `trim()` 之后添加 `if (dirPath.front() == L'"' && dirPath.back() == L'"') dirPath = dirPath.substr(1, len-2);`
2. 在调用 `BatchRename` 之前添加 `std::wcout << L"批量重命名指令已识别，目标: " << dirPath << std::endl;`
3. 执行完整构建验证无编译错误

## 验证方案
- 复制 `批量重命名 E:\test` → 应输出 "批量重命名指令已识别，目标: E:\test"
- 复制 `批量重命名 "E:\test with space"` → 应输出 "批量重命名指令已识别，目标: E:\test with space"
- 复制其他文本 → 不影响原有搜索功能

## 风险评估
- 仅添加 4 行代码，对原有逻辑无副作用
- 引号剥离仅在成功匹配 `批量重命名` 前缀后执行，不影响其他分支
- 回滚：直接删除对应代码行即可
